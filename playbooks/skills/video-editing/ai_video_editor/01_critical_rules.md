# Critical Rules — always-on rules across every module

These 14 numbered rules apply to EVERY scene and EVERY module. They were extracted from SKILL.md so the master index could stay small. **Load this module at the START of every run** (kickoff), and re-load it whenever a Gate Resume Scan tells you to.

The rules are intentionally redundant with the Per-Scene DoD (`ai_video_editor/03_dod_gates`) and the Jump-Prevention list (`ai_video_editor/02_jump_prevention`) — **redundancy is what stops misses**.

---

0. **Visual evidence gate — local-visible BEFORE reasoning.** Any image the agent uses as evidence (scene screenshots, AFTER screenshots, found media, B-roll thumbnails/contact sheets, downloaded photos, generated images, extracted spec images, style preset previews, reference images, **and cheap local overlay previews when available**) **must be saved/downloaded to a local file and SHOWN visibly to the user in chat BEFORE the agent looks at it, reasons from it, edits from it, uploads it, or advances.** For WideCast scene screenshots, there is exactly ONE valid transport: call MCP `scene_inspector` / `widecast_scene_inspector` with `action="screenshot_scene_280x498"`, read `result.screenshot.url`, run `curl -L -s -o <local>.jpg "<url>"`, then show the local file. The remote URL is only a short-lived transport; do NOT judge from it and do NOT embed it as evidence. If `result.screenshot.url` is missing, the screenshot step is not satisfied; do not use base64, binary `ImageContent`, sidecar JSON, browser screenshots, or REST-auth workarounds. Sidecar JSON alone (`returned_as`, `bytes`, `request_id`, or `status`) does NOT count as shown. Raw URLs, HTML with online `<img>`, or "I pulled a screenshot" are NOT enough. If the user cannot visibly see the exact local evidence image, the step is unchecked and the agent must not proceed. **Overlay preview exception:** a self-authored overlay does not require pre-upload local rendering when no cheap local viewer/converter is already available. Do not install tools, launch headless browsers, or probe multiple render paths just to make a preview; upload/apply the overlay and use the post-upload composite screenshot as the mandatory visual evidence.

1. **Name the field, never guess.** Selector is **`voice_file`** (not `id`). After every `modify_scene`, **pull `video_data`/`scene_geometry` again** to confirm it saved.

2. **Runtime = autonomous, end-to-end.** The user is NOT present; work scene 2 → last content scene in ONE pass, with the thumbnail synced immediately after scene 2, **never pause to ask**. Do not present options for the user to choose (`A or B?`, `which style?`, `should I rebuild?`). The agent must inspect the data + local-visible evidence, choose the best path under this playbook, act, and surface notes only in the final hand-off. (Per-scene pausing is dev-mode only.)

2a. **Decision protocol — choose, don't defer.** When several valid actions exist, decide by this priority: correctness of narration/data → face/subject preservation → readability at 280x498 → safe-zone/caption compliance → aesthetic quality/brand fit → minimal necessary edit. If two options are close, pick the simpler one that passes all gates. Only block for truly missing external input that no tool can infer, such as unavailable credentials or a required user-owned media file; otherwise continue autonomously.

2b. **WideCast edit trigger = full autonomous run.** If the user says anything like **"edit this video"**, **"finish this video"**, **"audit this video"**, **"process this video"**, **"review/fix this video"**, or equivalent in any language, and provides a WideCast review/editor URL or a WideCast `topic_id` (`widecast...`, `gubo...`, or a `record2.html#scene_editor?topic_id=...` link), treat that as explicit delegation to run the **complete WideCast video-editing workflow from beginning to final hand-off**. The default scope is always **Full audit + fix**: all content scenes, overlays, backgrounds/B-roll, narrator face clearance, thumbnail sync, final CTA, layout/composition, saved-state verification, and final hand-off. Do not ask what kind of edit, what to change, whether to proceed, which scene to start with, which style/background/overlay option to choose, or whether the agent has permission to edit. Do not use any clarification UI/tool for edit scope (for example a multiple-choice card, `request_user_input`, "Asking Edit scope", or options like `Full audit + fix / Specific scenes / Backgrounds / Text`). Do not say "Before I touch anything..." or "`edit this video` can mean different things" after a WideCast link/topic_id trigger. Pull `video_data`, infer the topic/context from the video itself, initialize the ledgers, run the full per-scene checklist, sync the thumbnail after scene 2, handle the final CTA, notify completion, and hand off the review URL. Only stop for a true blocker that tools/data cannot resolve (for example: invalid/inaccessible topic id, missing auth, unavailable required user-owned media, or a platform-required explicit render/export confirmation after editing).

3. **Decide by SIGHT, not by `pattern`.** grid-vs-real and regenerate-or-leave are visual calls from a local-shown BEFORE screenshot — the `pattern`/`sub_mode` labels are for the BLIND pipeline. **Vision gate:** if you cannot see images, do only safe data-driven layout fixes and leave overlays as-is. **If you CAN see, the screenshot is REQUIRED for every visual call — `scene_geometry` is NEVER a substitute for looking, and a scene you did not screenshot cannot be declared `PASS`. The image bytes are already in the `scene_inspector` result: save → show; never avoid the look because the bytes were awkward to extract.**

4. **Overlay = transparent internal vector, hosted, uploaded, then screenshot-verified.** Never paint a full-canvas background rect. Keep content in the safe box **x∈[36,684], y∈[128,960]**. Internally author/upload the overlay through the SVG pipeline, but **do not say "SVG", ".svg", "vector conversion", or `svg2spec` to normal users**; call it simply an **overlay** or **overlay preview**. Before hosting/uploading, save the authored overlay source locally. Show a local overlay preview only if the runtime can display it directly or convert it with an already-available lightweight path. If not, skip the pre-upload preview without mentioning format details, then `upload_asset`/`upload_overlay` and verify with the saved/shown local post-upload composite screenshot. Do **not** install tools, launch headless browsers, or spend time rasterizing/converting just to create a preview.

5. **ONE atom = ONE object** (`<g data-wc-object>`); atomize, never clump (list item, check, label each separate). To co-appear, share `data-wc-delay` — don't merge groups.

6. **Font: name the HEAVY family** (`"<Family> Black"`; cairosvg ignores numeric `font-weight`). Derive ONE font + accent **per video, vary between videos.** TITLE 60–96px Black; no overlay text < ~30px; fit each line inside its container.

6a. **Overlay copy correctness is its own gate.** Every visible overlay string — title, label, value, badge, card text, CTA, callout, chart label, handle, URL, currency, percent, proper noun — must be proofread against the full script, `quote`, `talking_point`, `visual`, and domain context. Typos, bad grammar, missing diacritics, wrong casing, wrong number/currency/symbol, wrong industry term, or awkward machine-written phrasing = Gate 4/7 FAIL even if layout, readability, and dead-zone checks pass. Caption/text correctness does NOT automatically prove overlay correctness because overlay text may be baked into `remotion_spec` or newly authored SVG.

7. **Diversify the LOOK — never ship flat-only.** Load the style library (`styles/text_axes` / `styles/chart_axes`) and reproduce a real look (gradient/glossy/3D/metallic/bevel/…), one per video.

8. **A-roll: the face is sacred.** Never edit `narrator_face`; solve narrator + overlay + caption together. The current overlay position/size is not fixed evidence: a full-canvas narrator trial cannot fail just because the old overlay touches the face, caption, or dead zone. Move/resize/simplify/rebuild the overlay first; shrink the narrator only after higher-priority joint compositions fail. Ignore face overlap on B-roll. **Final A-roll / CTA scene special rule:** if the scene is the last non-thumbnail/content scene and `show_narrator=true` / `active_roll="A"`, load `40_thumbnail_cta`, treat the narrator as the primary closing element, and use one clear typography-led CTA when the scene asks for action. Prefer a full-canvas narrator first, then shifted-full-canvas only if a small CTA support overlay needs room. Do not replace the human close with object clutter or a large checklist/chart unless a detail-dense visual is truly indispensable. No-overlay is acceptable only when the visible narrator + caption already carry the CTA clearly; otherwise use a short bold lower-third/chest/side CTA that clears the face and caption.

9. **Grid ≤ 3 scenes/video, all sharing ONE grid.** Lean to a real background otherwise.

10. **Realistic photos are REUSED, never "drawn".** To ONLY add info to a good overlay use `modify_scene` (M) `remotion.add_element` (additive, no overwrite); full re-author overwrites the spec.

11. **Show found media in chat BEFORE looking/evaluating/applying** (local file evidence — the sandbox can't fetch online URLs in a widget). No private preview first.

12. **Every scene = an overlay review AND a BACKGROUND audit — two separate passes; never skip the background** (it is its own gate, not part of the overlay check). Gate 4 handles overlay first. Gate 5 then audits the background using TWO local-visible evidence images: the BEFORE composite screenshot (render truth) AND the active background/media plate (`thumbnailUrl`/fallback media thumbnail) shown separately so the agent can tell which marks belong to the background vs the overlay/caption. Placing Gate 5 immediately after overlay is deliberate: after the long overlay workstream, the next checklist item must pull the agent back to the background audit. And **every screenshot/media plate you pull is saved/downloaded locally and SHOWN visibly to the user before you act on it**. **No visible local evidence = no visual judgment.** Complete the per-scene Definition of Done (`ai_video_editor/03_dod_gates`) before advancing to the next scene.

12a. **Endpoint scenes: scene 2 + thumbnail + final CTA.** The first real scene after the thumbnail (usually `id=2`) is special regardless of `type`/`pattern`: load `40_thumbnail_cta` and treat its first-frame composition like a thumbnail/poster, because platforms may auto-extract that frame as the default cover. Use a short whole-video hook/consequence title, poster-thick typography, strong face/subject preservation, and controlled overlay decoration. It must feel like dynamic poster typography / magazine-cover thumbnail / another named endpoint style, not a normal centered card, horizontal text bar, or inside-scene title panel. **Immediately after scene 2 PASS, before starting scene 3, sync the static thumbnail scene:** apply the same uploaded poster overlay URL to `type="thumbnail"` by `voice_file`, pull/show the thumbnail screenshot, and confirm it saved with `video_data`/`scene_geometry`. This sync is a gate; do not continue to scene 3 until it is done. **Do not re-check thumbnail at the end** unless the user explicitly asks; the thumbnail and scene 2 should already be intentionally matched. When reaching the last non-thumbnail/content scene, especially `type="CALL TO ACTION"`, load `40_thumbnail_cta` again and treat it as a closing CTA endpoint: one clear action, typography stronger than objects, narrator-primary if A-roll, and poster-grade composition rather than normal explainer-card composition.

12b. **Gate 5 requires a printed BACKGROUND PROOF, not a vibe check.** The agent must print/fill the exact `Gate 5 BACKGROUND PROOF` template in `ai_video_editor/03_dod_gates` for every content scene after Gate 4 overlay work and before Gate 6 final composition starts. A scene cannot be called PASS, and the run cannot be called done, unless every content scene has a `Gate 5 BACKGROUND PROOF` row with `Verdict: PASS keep`, `Verdict: PASS grid-by-design`, or `Verdict: FIXED + PASS`. If the proof is missing, answer "background audit not complete" and run Gate 5; do not summarize the scene as done.

12c. **Attention-drift / Gate Resume Scan — a detour is NOT a completed scene or run.** Any issue can pull the agent away from the original checklist: wrong term, typo, missing number/symbol, background replacement, geo/currency mismatch, overlay rebuild, covered face, caption collision, thumbnail sync, tool/debug problem, or a small fix applied across many scenes. After the detour is fixed, run a **Gate Resume Scan** before summarizing or moving on: re-open the current scene checklist, identify the earliest unchecked or invalidated gate, and continue from there. If the detour touched multiple scenes, resume at the earliest scene/gate whose PASS is no longer proven. No gate is exempt: Gate 1 text, Gate 2 role, Gate 3 BEFORE evidence, Gate 4 overlay proof, Gate 5 background proof, Gate 6 final composition/dead-zone, Gate 7 AFTER evidence, Gate 8 server-saved confirmation, and Gate 9 module coverage can all be missed after drift.

12d. **Pre-summary completion scan — no big task left behind.** Before writing any final summary, completion report, Telegram completion message, or export question, scan the run-level checklist, not memory: every content scene must have `Scene N: PASS`; every scene must have Gate 1–9 checked; every scene/run must have a passing **MODULE COVERAGE GATE**; every Background Audit Ledger row must be filled; scene 2 thumbnail sync must be complete; final CTA endpoint handling must be complete for the last content scene; any `[ACTION REQUIRED]` item must be surfaced. If any major workstream is incomplete (especially background audit/replacement, endpoint scene handling, module loading, or server-saved verification), do the missing work first. Fixing a small defect across all scenes does NOT mean the video is done.

13. **The master SKILL.md is an INDEX — load the named module BEFORE doing each step, every time** (a required ACTION, see SKILL.md "HOW TO USE"). You cannot do a step from the index alone; not opening the module = skipping that step's rules. (This is the literal cause of a missed background audit: the agent never opened `20_background`.) Re-load it each time you reach the step, even if loaded earlier this session; a resumed run reloads, never works from memory.

13a. **Module Coverage Gate — missing playbook = not done.** Gate 9 requires the agent to prove the required playbooks were loaded for the run/scene: `00_ENTRYPOINT`, `01_critical_rules`, `02_jump_prevention`, `03_dod_gates`, `04_principles_workflow`, `05_quality_qa_priority`, `10_mechanics`, `20_background`, `30_overlay_core`, `31_typography`, `32_charts`, `33_patterns`, and `40_thumbnail_cta` when applicable. Missing a required file means the scene/video is incomplete. But loading a file once is not completion: Gate 9 only proves coverage; each gate still requires its own evidence/proof/verdict.

14. **Announce the per-scene task list at the START, report progress at each gate, and re-state the ✓/✗ checklist at the END** (`ai_video_editor/03_dod_gates`) — so the user can audit your plan, your current position, and any gaps. Announce, don't pause. **End every scene with an explicit `Scene N: PASS`/`FAIL` verdict — say PASS only after scanning all 9 DoD gates + §7 (`ai_video_editor/05_quality_qa_priority`) and confirming each; never advance without a stated PASS.**

## SELF-AUDIT — run this BEFORE every reply (cheap meta-gate)

Before sending any reply, silently confirm — and fix any "no" before replying:

- Did I open the module(s) for the action I'm taking THIS turn — via `widecast_get_editing_skill(module='<id>')` (MCP transport) or local `Read("<id>.md")` (Anthropic Skill upload transport) — and NOT work from the index / from memory?
- Every image I used as evidence — did I save it locally AND show it to the user before I judged from it?
- Am I making visual calls from a SCREENSHOT, not from `scene_geometry` alone?
- Am I on the per-scene DoD — which gate am I on, and did I announce progress?
- If I'm ending a scene: did I scan the checklist + §7 and state `Scene N: PASS` / `FAIL`?
- Did I run the `MODULE COVERAGE GATE`, and does it show no missing required playbook?
- Did I avoid jumping any gate (background pass · overlay LOAD CHAIN · dead-zone · verdict-before-next-scene)?
- Field discipline: `voice_file` selector, and did I re-pull to confirm the edit saved?
- If I just finished any detour/fix/debug path: did I run the Gate Resume Scan and continue from the earliest unchecked or invalidated gate instead of summarizing?
- If I am about to summarize/hand off/export/notify completion: did I run the Pre-summary completion scan, and are all scenes PASS with a complete Module Coverage Gate, Background Audit Ledger, scene 2 thumbnail sync, and final CTA endpoint handling?
- If the human must act, did I use a standalone `[ACTION REQUIRED]` block?

This list is intentionally redundant with the Critical Rules above, the Jump-Prevention list (`ai_video_editor/02_jump_prevention`), and the per-scene DoD (`ai_video_editor/03_dod_gates`) — **the redundancy is the point** (it is why misses are rare).
